# OA 電子表單草稿、填寫模板與表單申請權限設計需求

> 文件用途：提供 Claude Code 開發 OA 電子表單功能時參考  
> 適用範圍：電子表單發起申請、我的申請、我的草稿、我的填寫模板、表單權限顯示、表單分類  
> 文件定位：本文件只描述產品規則、功能邏輯與 UI 調整方向，不包含資料表設計

---

## 1. 設計背景

目前電子表單頁面中出現了「臨時保存」與「模板保存」兩個按鈕，但兩者概念容易混淆。

實際業務上需要區分三種不同功能：

```text
草稿：使用者填表填到一半，尚未送出，可以先保存，之後回來繼續編輯。
填寫模板：使用者將常用填寫內容保存起來，之後建立同類表單時快速帶入。
表單申請權限：不同員工只能看到與申請自己有權限的電子表單。
```

因此需要調整電子表單功能設計與選單結構，避免使用者混淆，也讓後台維護更簡單。

---

## 2. 名詞定義

### 2.1 草稿

草稿代表一張尚未送出的電子表單。

草稿特性：

```text
尚未送出
尚未產生正式申請單號
尚未進入審批流程
尚未產生審批快照
可由申請人繼續編輯
可刪除
送出後不再是草稿
```

草稿的主要用途是：

```text
使用者填表填到一半，可以先保存，之後再回來繼續填寫。
```

---

### 2.2 填寫模板

填寫模板代表使用者保存的常用填寫內容。

填寫模板不是一張申請單，也不是審批模板。

填寫模板特性：

```text
用於重複填寫類似表單時快速帶入資料
不會進入審批流程
不會產生申請單號
不保存審批線
不保存審批人
不保存審批快照
不保存表單狀態
```

填寫模板的主要用途是：

```text
讓使用者將例行性、重複性的欄位內容保存起來，下次建立新申請時快速套用。
```

例如：

```text
GSuite 月費申請
雲端服務費用申請
固定伺服器費用申請
固定資訊需求申請
常用 OA 資產申請
```

---

### 2.3 表單模板、審批模板、填寫模板的差異

系統中可能同時存在三種模板，必須明確區分。

```text
表單模板：定義這張電子表單有哪些欄位，由系統 / 表單管理員維護。
審批模板：定義這張表單要走哪些審批節點，由管理員維護。
填寫模板：保存使用者常用填寫內容，由使用者或授權人員維護。
```

填寫模板不可取代表單模板，也不可取代審批模板。

## 3. 左側選單建議

電子表單底下建議調整為：

```text
電子表單
├── 發起申請
├── 我的申請
├── 我的草稿
└── 我的填寫模板
```

若系統已有獨立「簽核中心」，則不要把待簽核功能放在電子表單底下，避免功能重複。

建議整體結構：

```text
電子表單
├── 發起申請
├── 我的申請
├── 我的草稿
└── 我的填寫模板

簽核中心
├── 待我簽核
├── 我已簽核
├── 抄送給我
└── 簽核查詢
```

設計原則：

```text
電子表單 = 我發起的申請
簽核中心 = 我要處理別人的申請
```

## 4. 發起申請頁設計

「發起申請」頁是讓使用者選擇要建立哪一張電子表單。

此頁面不應出現：

```text
臨時保存
模板保存
儲存草稿
另存為填寫模板
```

原因是使用者尚未進入具體表單，也尚未填寫內容，因此沒有可保存的草稿或模板。

發起申請頁應提供：

```text
搜尋表單
分類篩選
常用表單
最近使用
表單卡片列表
```

建議頁面內容：

```text
發起申請
├── 搜尋表單
├── 常用表單
├── 最近使用
├── 分類：全部 / HR / 財務 / 行政 / PMO / 商務 / 合同
└── 表單卡片
```

---

## 5. 表單填寫頁按鈕設計

當使用者進入某一張具體表單填寫頁後，才應顯示保存相關按鈕。

建議按鈕：

```text
儲存草稿
另存為填寫模板
送出申請
```

不要使用：

```text
臨時保存
模板保存
```

原因：

```text
臨時保存不夠明確，使用者不容易理解它其實是草稿。
模板保存容易和系統表單模板、審批模板混淆。
```

建議命名：

```text
儲存草稿
另存為填寫模板
```

或較口語版本：

```text
儲存草稿
存成常用模板
```

本系統建議採用較清楚的正式命名：

```text
儲存草稿
另存為填寫模板
```

---

## 6. 草稿功能規則

### 6.1 儲存草稿

使用者在表單填寫頁可點擊「儲存草稿」。

儲存草稿後：

```text
表單資料保存為草稿
不送出審批
不產生審批流程
不產生審批快照
不通知審批人
使用者可在「我的草稿」中繼續編輯
```

---

### 6.2 我的草稿

「我的草稿」應列出目前登入者尚未送出的表單。

草稿列表建議顯示：

```text
表單名稱
草稿標題 / 申請主旨
建立時間
最後保存時間
表單分類
操作：繼續編輯 / 刪除
```

草稿操作：

```text
繼續編輯
刪除草稿
送出申請
```

---

### 6.3 草稿送出

草稿送出時，必須執行與新表單送出相同的檢查：

```text
必填欄位驗證
欄位格式驗證
附件要求驗證
表單申請權限驗證
審批模板匹配
審批人解析
審批流程快照產生
```

草稿保存時不需要完整審批驗證，但送出時必須完整驗證。

---

## 7. 填寫模板功能規則

### 7.1 另存為填寫模板

使用者在表單填寫頁可點擊「另存為填寫模板」。

系統應讓使用者輸入：

```text
模板名稱
模板說明，可選
是否設為常用，可選
```

填寫模板保存的是目前表單中可重複使用的欄位內容。

---

### 7.2 填寫模板不應保存的內容

填寫模板不得保存以下內容：

```text
審批線
審批人
審批快照
審批狀態
申請單號
簽核紀錄
送出時間
審批完成時間
已核銷狀態
上次發票號碼
上次 invoice 編號
上次實際付款日期
上次附件憑證
```

原因：

```text
填寫模板只用於快速填寫，不應影響正式審批流程。
審批流程必須在每次送出時，依當下最新規則重新計算。
```

---

### 7.3 填寫模板可保存的內容

填寫模板可保存重複性高、低風險、與審批狀態無關的欄位。

例如：

```text
申請類別
費用類型
供應商名稱
常用用途說明
常用設備規格
常用需求描述
分攤項目設定
分攤比例設定
常用收款資訊
附件要求說明
```

實際可保存欄位可依表單類型設定。

---

### 7.4 套用填寫模板

使用者建立新表單時，可以選擇「從填寫模板建立」。

套用流程：

```text
選擇表單類型
選擇填寫模板
系統帶入模板中的可重複欄位
系統清空或提示確認當次申請欄位
使用者補齊本次申請資料
送出申請
系統重新驗證與重新解析審批流程
```

---

### 7.5 套用模板後需提醒使用者確認

使用者套用模板後，系統應提示：

```text
已套用填寫模板，請確認金額、月份、附件、分攤項目與申請內容是否正確。
```

對於費用類表單，以下欄位建議清空或標記為需要確認：

```text
申請月份
實際金額
發票號碼
invoice 編號
附件
付款日期
本次備註
```

---

## 8. 從歷史申請複製

除了填寫模板，也建議支援「從歷史申請複製」。

使用場景：

```text
使用者想複製上個月的 GSuite 費用申請。
使用者想複製上次的資訊需求申請。
使用者想快速建立類似的 OA 資產申請。
```

操作入口：

```text
我的申請
→ 找到歷史申請
→ 點擊「複製建立新申請」
```

複製規則：

```text
只複製可重用欄位
不複製審批線
不複製審批人
不複製審批快照
不複製表單狀態
不複製申請單號
不複製簽核紀錄
不複製已核銷狀態
附件預設不複製
```

從歷史申請複製後，仍需在送出時重新驗證與重新解析審批流程。

---

## 9. 我的填寫模板

「我的填寫模板」用於管理目前登入者建立的填寫模板。

列表建議顯示：

```text
模板名稱
表單類型
表單分類
最近使用時間
建立時間
是否常用
是否啟用
操作：使用 / 編輯 / 複製 / 刪除
```

支援操作：

```text
從模板建立申請
編輯模板名稱
編輯模板內容
複製模板
刪除模板
設為常用 / 取消常用
啟用 / 停用
```

---

## 10. 共用填寫模板

第一版可以先只做個人填寫模板。

若要支援共用模板，建議作為第二階段功能。

共用模板適用情境：

```text
財務提供常用費用申請模板
行政提供常用資產申請模板
IT 提供常用資訊需求模板
HR 提供常用人事申請模板
```

共用範圍可支援：

```text
指定公司
指定部門
指定項目
指定角色
全公司
全集團
```

共用模板應由有權限的人維護，一般員工不應任意建立全集團共用模板。

---

## 11. 表單分類與權限顯示

電子表單分類只是幫助使用者找表單。

真正控制使用者能不能看到與申請某張表單的，應該是表單申請權限。

畫面上的分類例如：

```text
全部
商務
合同
財務
行政
PMO
HR
```

應只顯示目前登入者有權限申請的表單。

---

## 12. 「全部」分類顯示規則

「全部」不是顯示系統內全部表單，而是：

```text
目前登入使用者有權限申請的全部表單。
```

例如一般員工登入後可能看到：

```text
請假申請
補卡申請
加班申請
誤餐費申請
資訊需求申請
OA 資產申請
```

PMO 人員登入後可能看到：

```text
請假申請
補卡申請
項目立項申請
項目變更申請
項目終止申請
```

分類底下若沒有任何可申請表單，前端可不顯示該分類。

---

## 13. 表單申請權限設計原則

公司人員數量較多，不應逐一為每位員工設定可申請表單。

建議使用以下方式管理：

```text
角色權限
組織範圍
個人例外
```

核心原則：

```text
常態權限用角色或組織設定。
特殊情況才用個人例外。
不要用個人名單管理大量權限。
```

---

## 14. 表單可申請對象

每一張電子表單都應能設定可申請對象。

第一版建議支援：

```text
全體員工
指定公司
指定部門
指定系統角色
指定人員例外
```

後續可擴充：

```text
指定事業部
指定項目
指定職級
指定職位
指定審批角色
```

---

## 15. 權限判斷邏輯

第一版建議採用簡單 OR 邏輯：

```text
使用者符合任一可申請條件 → 可看到並申請該表單
使用者不符合任何條件 → 不顯示該表單
```

例如：

```text
項目立項申請
可申請對象：
- 角色 = PMO
- 角色 = 事業部主管
- 指定人員 = 特定高管

使用者符合任一條件，即可申請。
```


```text

可申請對象：主管角色、HR 角色、指定人員例外
```


## 17. 個人例外使用規則

個人例外只應用於少數特殊情境。

例如：

```text
某位高管沒有 PMO 角色，但可以申請項目立項。
某位行政助理需要代主管填寫特定表單。
某位臨時負責人短期需要申請某類表單。
```

不應使用個人例外管理大量常態權限。

常態權限應透過：

```text
角色
公司
部門
職位
組織範圍
```

---

## 18. 表單管理後台建議

系統應提供電子表單管理設定。

每張表單至少可設定：

```text
表單名稱
表單分類
是否啟用
可申請對象
可見對象
是否允許儲存草稿
是否允許另存為填寫模板
是否允許從歷史申請複製
預設審批模板
排序
```

其中「可申請對象」應支援：

```text
全體員工
指定公司
指定部門
指定角色
指定人員例外
```

## 19. 常用表單與最近使用

為了改善使用者體驗，發起申請頁建議支援：

```text
常用表單
最近使用
收藏表單
```

### 19.1 常用表單

可依系統統計或管理員設定顯示常用表單。

例如：

```text
請假申請
補卡申請
加班申請
資訊需求申請
費用報銷
```

### 19.2 最近使用

顯示使用者最近建立過的表單類型。

### 19.3 收藏表單

使用者可將常用表單加入收藏，方便下次快速發起。

## 20. 表單列表頁 UI 調整建議

目前畫面右上角的：

```text
臨時保存
模板保存
```

不應出現在發起申請列表頁。

原因：

```text
發起申請列表頁只是選擇表單，尚未填寫任何內容。
沒有草稿可以儲存，也沒有填寫內容可以保存為模板。
```

建議調整為：

```text
發起申請列表頁：
- 搜尋表單
- 分類篩選
- 常用表單
- 最近使用
- 表單卡片

表單填寫頁：
- 儲存草稿
- 另存為填寫模板
- 送出申請
```

---

## 21. 表單填寫頁 UI 建議

表單填寫頁應清楚顯示目前狀態。

若從草稿進入：

```text
草稿最後保存時間：YYYY-MM-DD HH:mm
```

若從填寫模板建立：

```text
已套用填寫模板：GSuite 月費申請
```

若從歷史申請複製：

```text
已從歷史申請複製，請確認本次申請資料。
```

送出前應提醒：

```text
送出後將依目前表單內容重新產生審批流程，草稿或填寫模板不會保存審批線。
```

---

## 22. 第一版建議實作範圍

第一版建議優先實作：

```text
1. 發起申請頁只顯示使用者有權限申請的表單。
2. 表單列表頁移除「臨時保存」與「模板保存」按鈕。
3. 表單填寫頁提供「儲存草稿」。
4. 電子表單選單增加「我的草稿」。
5. 表單填寫頁提供「另存為填寫模板」。
6. 電子表單選單增加「我的填寫模板」。
7. 我的申請支援「複製建立新申請」。
8. 填寫模板不保存審批線與表單狀態。
9. 草稿送出時重新驗證並重新解析審批流程。
10. 表單權限用角色、公司、部門、個人例外管理，不逐人設定大量權限。
```

第二階段可再擴充：

```text
共用填寫模板
收藏表單
最近使用
更細的組織範圍權限
模板版本提示
模板套用欄位選擇
```

## 23. 開發注意事項

Claude Code 開發時請注意：

```text
1. 草稿是未送出的申請單，不是模板。
2. 填寫模板是常用填寫內容，不是表單模板，也不是審批模板。
3. 填寫模板不可保存審批線、審批人、審批快照與審批狀態。
4. 從草稿、模板、歷史申請送出時，都必須重新跑欄位驗證與審批流程解析。
5. 發起申請頁只顯示目前登入使用者有權限申請的表單。
6. 表單分類只是顯示輔助，真正權限綁在表單本身。
7. 全部分類只顯示目前登入者可申請的全部表單。
8. 不要為 400 多名員工逐一設定表單權限，應使用角色、公司、部門與少量個人例外。
9. 表單列表頁不應提供草稿保存與模板保存，這些功能應放在具體表單填寫頁。
10. UI 命名請使用「儲存草稿」與「另存為填寫模板」，不要使用「臨時保存」與「模板保存」。
```

## 24. 最終設計結論

本功能應拆成三個清楚概念：

```text
草稿：保存未送出的申請單。
填寫模板：保存常用填寫內容。
表單申請權限：控制誰可以看到與申請哪些表單。
```

最佳設計：

```text
電子表單
├── 發起申請：只顯示我有權限申請的表單
├── 我的申請：查看已送出的申請
├── 我的草稿：查看與編輯未送出的申請
└── 我的填寫模板：管理常用填寫內容
```

按鈕命名：

```text
臨時保存 → 儲存草稿
模板保存 → 另存為填寫模板
```

權限管理：

```text
不要逐人設定大量權限。
使用角色、公司、部門、組織範圍與少量個人例外。
```

送出規則：

```text
無論從草稿、填寫模板或歷史申請建立，送出時都必須重新驗證欄位、重新匹配審批模板、重新解析審批人並產生新的審批快照。
```
