# OA 系統審批角色、審批對應與審批規則設計需求

> 文件用途：定義 OA 系統審批流的角色語意、審批對應關聯設計與實作規則  
> 適用範圍：電子表單、人事假勤、財務申請、行政申請、簽核中心、審批模板設定  
> 文件狀態：**已實作（v2）**

---

## 1. 設計背景

OA 系統的審批流設計遵守以下核心原則：

```text
不把審批人直接寫死在表單或員工資料上。
不把員工帳號直接綁定為某個審批角色作為主要設計。
不把系統角色（HR_ADMIN、FINANCE_ADMIN 等）直接當審批角色使用。
不讓每種表單在程式碼中各自寫一套審批邏輯。
```

正確做法：

```text
審批角色  →  定義流程節點的語意（「這一關要找哪一類人？」）
審批對應  →  指定每個「分組 + 角色」由哪位人員負責（有且只有一位預設）
審批模板  →  定義某種表單走哪些審批節點（引用審批角色代碼）
審批快照  →  表單送出後固定當下解析出的實際流程
```

### 分組定義

**分組**為公司、事業部、項目、部門的統稱。每個分組下可為每種審批角色指定多位人員，其中必須且只能有一位設為**預設**，系統送出表單時取預設人員作為審批人。

| 分組類型 | 說明 |
|---------|------|
| 公司 | 法人公司層級 |
| 事業部 | 公司下的事業部 |
| 項目 | 事業部或公司下的項目 |
| 部門 | 公司下的部門 |
| 特殊 | 跨分組的特殊角色（執行長、董事長等集團層級職務） |

## 2. 審批角色分類（共兩類）

### 2.1 動態組織解析型

這類角色直接從組織架構資料中解析，**不需要設定審批對應**。

| 角色代碼 | 中文名稱 | 解析來源 |
|---------|---------|---------|
| `applicant_direct_manager` | 申請人直屬主管 | `user_org_assignments.directManagerUserId` |
| `department_manager` | 部門主管 | `Department.managerUserId` |

### 2.2 審批對應解析型

這類角色透過「審批對應關聯表」解析，需要在帳號管理或分組管理中設定對應人員。設定審批步驟時，需同時指定**分組類型**與**角色**。

| 角色代碼 | 中文名稱 | 說明 |
|---------|---------|------|
| `lead` | 負責人 | 依步驟設定的分組類型解析對應負責人 |
| `hr` | 人事專員 | 依步驟設定的分組類型解析對應人事專員 |
| `hr_manager` | 人事主管 | 依步驟設定的分組類型解析對應人事主管 |
| `finance` | 財務人員 | 依步驟設定的分組類型解析對應財務人員 |
| `finance_manager` | 財務主管 | 依步驟設定的分組類型解析對應財務主管 |
| `ceo` | 執行長 | 特殊層級 |
| `chairman` | 董事長 | 特殊層級 |

> `ceo` / `chairman` 固定為「特殊」分組類型，設定步驟時分組固定，不可更改。

---

## 3. 審批對應關聯表設計

### 3.1 資料結構

每筆記錄代表「某分組 + 某角色 → 某人員」的對應，並標記是否為預設審批人。

| 欄位 | 說明 |
|------|------|
| `scopeType` | 分組類型：`company` / `business_unit` / `project` / `department` / `special` |
| `scopeId` | 分組實體 ID（UUID，`special` 時為 null） |
| `roleCode` | 審批角色代碼：`lead` / `hr` / `hr_manager` / `finance` / `finance_manager` / `ceo` / `chairman` |
| `userId` | 負責人員（`users.id`） |
| `isDefault` | 是否為預設審批人 |

### 3.2 預設規則

- 每個「`scopeType` + `scopeId` + `roleCode`」組合，**必須且只能有一位** `isDefault = true`
- 新增第一位人員時，自動設為預設
- 移除預設人員後，需手動指定新的預設（或刪除最後一位後該組合無預設警告）
- 在帳號管理或分組管理中均可點擊切換預設（需二次確認）

### 3.3 解析邏輯

送出表單時，系統依審批步驟的 `scopeConfig.groupType` 以及申請人所屬的組織資料，查詢對應的預設人員：

```
scopeType = groupType（步驟設定）
scopeId   = 申請人所屬的對應分組 ID
roleCode  = approverType（步驟設定）
isDefault = true
```

特殊層級（ceo / chairman）查詢條件為 `scopeType = 'special'`，`scopeId = null`。

## 4. 員工資料設計原則

員工資料中**不應直接標記審批功能角色**。

員工身上只記錄組織歸屬資料：
```
所屬地區、所屬公司、所屬事業部、所屬項目、所屬部門、直屬主管、職位、職級
```

審批責任透過審批對應關聯表設定，員工作為某分組+角色的負責人員。

## 5. 審批模板步驟設計

### 5.1 步驟與串並簽

- **串簽**：`approvalMode = 'any'`（任一人通過即可）或步驟只有一位審批人
- **並簽**：`approvalMode = 'all'`（所有審批人都需簽核，通知所有人）
- **多數通過**：`approvalMode = 'majority'`

### 5.2 審批對應解析型步驟的分組設定（scopeConfig）

對於**審批對應解析型**角色，步驟需設定**分組類型**，決定按照哪個維度查詢對應人員：

```json
// 分組類型：公司
{ "groupType": "company", "groupResolution": "applicant_company" }

// 分組類型：事業部（申請人所屬事業部）
{ "groupType": "business_unit", "groupResolution": "applicant_bu" }

// 分組類型：事業部（指定特定事業部）
{ "groupType": "business_unit", "groupResolution": "specific_bu", "businessUnitId": "uuid" }

// 分組類型：項目（申請人所屬項目）
{ "groupType": "project", "groupResolution": "applicant_project" }

// 分組類型：項目（指定特定項目）
{ "groupType": "project", "groupResolution": "specific_project", "projectId": "uuid" }

// 分組類型：部門（申請人所屬部門）
{ "groupType": "department", "groupResolution": "applicant_department" }

// 分組類型：特殊（ceo / chairman 固定使用）
{ "groupType": "special" }
```

## 6. 多模板匹配（金額條件）

```
採購申請 - 低金額（≤ 10,000）
  步驟：部門主管 → 公司・財務人員

採購申請 - 中金額（10,000 ~ 100,000）
  步驟：部門主管 → 項目・負責人 → 公司・財務主管

採購申請 - 高金額（> 100,000）
  步驟：部門主管 → 項目・負責人 → 事業部・負責人 → 特殊・執行長 → 特殊・董事長
```

---

## 7. 送出前驗證規則

任何表單送出前，後端必須驗證（`POST /api/approvals/validate-form`）：

1. 是否有符合條件（formType + 金額範圍）的啟用模板
2. 每個必要步驟的審批角色是否能解析出審批人：
   - 審批對應解析型：該分組+角色是否有設定預設人員（`isDefault=true`）
   - 動態解析型：直屬主管是否已設定（部門主管在部門資料中驗證）
   - 指定人員型：是否已選擇人員

---

## 8. 每種表單必有預設審批線

每一種可以送出的電子表單（form_type），至少需要一條 `isActive = true` 的審批模板。

---

## 9. 審批快照規則

表單送出後，系統固定當下解析出的實際審批流程（`approval_instances` + `approval_steps`）。

後續以下變更不影響已送出的表單：
- 審批模板修改 / 步驟變更
- 審批對應異動（更換預設人員）
- 員工組織歸屬異動

駁回後重新送出時，需重新解析當下最新設定，產生新的審批快照。

## 10. UI 操作入口

### 10.1 審批流設定（系統管理員 → 審批流設定）

```
審批流設定
├── Tab 1: 審批模板   ← 建立/編輯審批模板與步驟
└── Tab 2: 審批角色   ← 查看所有角色定義與解析方式（唯讀說明）
```

### 10.2 審批步驟的審批人設定

**① 動態組織解析**（無需額外設定）

| 選項 | 說明 |
|------|------|
| 申請人直屬主管 | 從員工組織配置解析 |
| 部門主管 | 從部門資料解析 |

**② 分組 + 角色**（審批對應解析型）

| 分組（必選） | 可選角色 |
|------------|---------|
| 公司 | 負責人 / 人事專員 / 人事主管 / 財務人員 / 財務主管 |
| 事業部 | 負責人 / 人事專員 / 人事主管 / 財務人員 / 財務主管 |
| 項目 | 負責人 / 人事專員 / 人事主管 / 財務人員 / 財務主管 |
| 部門 | 負責人 / 人事專員 / 人事主管 / 財務人員 / 財務主管 |
| 特殊 | 執行長 / 董事長 |

**③ 指定人員**

直接選擇特定員工帳號。

### 10.3 帳號管理（系統管理員 → 帳號管理 → 編輯）

帳號編輯頁新增「審批職能」區塊：

```
審批職能
├── [+ 新增審批職能]
└── 角色清單（依 roleCode 分組）
    └── 每個角色下：已指派的分組列表
        ├── 分組名稱 | 是否預設（標籤）| [設為預設] [移除]
        └── ...
```

- 點擊「設為預設」需二次確認，確認後更新該分組+角色下的預設人員
- 新增時選擇：角色 + 分組類型 + 具體分組

### 10.4 分組管理（各分組編輯頁）

公司、事業部、項目、部門的編輯對話框新增「審批職能」分頁：

```
審批職能
├── [+ 新增審批職能]
└── 角色清單（依 roleCode 分組）
    └── 每個角色下：已指派的人員列表
        ├── 人員姓名 | 是否預設（標籤）| [設為預設] [移除]
        └── ...
```

## 11. 資料庫設計

| 表名 | 說明 |
|------|------|
| `approval_assignments` | 審批對應關聯表（分組 + 角色 + 人員 + 是否預設） |
| `approval_templates` | 審批模板（formType、金額範圍、優先序）|
| `approval_template_steps` | 模板步驟（順序、串/並簽模式、允許加簽）|
| `approval_template_step_approvers` | 步驟審批人（角色代碼、scopeConfig、指定人員）|
| `approval_instances` | 審批快照（表單送出後生成）|
| `approval_steps` | 步驟快照（含動態加簽步驟）|
| `approval_step_approvers` | 步驟審批人快照（記錄實際解析人員）|

---

## 12. 後端 API 清單

```
GET    /api/approvals/assignments               查詢審批對應（?scopeType=&scopeId=&roleCode=&userId=）
POST   /api/approvals/assignments               新增對應（自動設第一筆為預設）
DELETE /api/approvals/assignments/:id           移除對應
PATCH  /api/approvals/assignments/:id/default   設為預設（清除同分組+角色下其他預設）

GET    /api/approvals/templates                 列出審批模板
POST   /api/approvals/templates                 新增審批模板
PATCH  /api/approvals/templates/:id             更新審批模板
PUT    /api/approvals/templates/:id/steps       替換步驟

GET    /api/approvals/instances/:id             取得審批快照詳情
POST   /api/approvals/instances/:id/approve     審批通過
POST   /api/approvals/instances/:id/reject      審批駁回

POST   /api/approvals/validate-form             送出前驗證審批流程完整性
```
