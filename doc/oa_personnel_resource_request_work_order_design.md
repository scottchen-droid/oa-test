# OA 人員資源申請單與工作單中心設計需求

> 文件用途：提供 Claude Code 依據本需求補充 OA 系統中的「人員資源申請」與「工作單中心」功能。  
> 適用範圍：人員入職、離職、在職資源新增 / 變更 / 回收、帳號開通、門禁卡、指紋、電話、Apple ID、Jira、Mattermost 等資源處理。  
> 文件定位：本文件描述產品規則與流程邏輯，不包含資料表設計。

---

## 1. 設計背景

目前人員入職、離職或在職期間，會需要申請或回收多種公司資源，例如：

- Mattermost 帳號
- Jira 帳號
- 門禁卡
- 電話 / 分機
- 指紋設定
- Apple ID
- VPN
- Google Workspace
- 其他公司內部系統帳號或實體資源

這些資源的前段審批邏輯通常相似，例如都需要直屬主管與人事主管確認，但後續負責處理的單位不同。

例如：

```text
Mattermost：IT 處理
Jira：PMO / IT 處理
門禁卡：行政處理
指紋設定：行政 / HR 處理
電話：行政 / 總務處理
Apple ID：IT / 資產管理處理
```

若每一種資源都獨立建立一張電子表單，會造成：

```text
1. HR 需要重複開很多張單。
2. 主管需要重複審批類似內容。
3. 審批線大量重複。
4. 後續追蹤困難。
5. 離職回收時容易漏掉帳號或資源。
6. 系統維護成本提高。
```

因此，本需求建議不要為每個資源建立獨立電子表單，而是使用一張「人員資源申請單」承接多種資源需求，並在審批通過後由系統自動產生各單位的工作單。

---

## 2. 核心設計原則

本功能的核心原則為：

```text
電子表單負責申請與審批。
工作單負責後續執行與回填結果。
```

也就是：

```text
一張人員資源申請單
→ 勾選多個資源項目
→ 主管 / 人事主管審批
→ 審批通過後產生多張工作單
→ 各負責單位處理自己的工作單
→ 工作單完成後回填帳號 / 卡號 / 分機等結果
→ 結果綁定回人員資料
→ 全部工作單完成後主單結案
```

系統不應將 IT、PMO、行政等後勤單位全部設計成審批人。這些單位在大多數場景中不是判斷「是否同意」，而是負責「執行工作」。

---

## 3. 電子表單設計：人員資源申請單

系統需要建立一張電子表單：

```text
人員資源申請單
```

這張表單用來處理以下場景：

```text
1. 入職資源開通
2. 在職資源新增
3. 在職資源變更
4. 離職資源回收
```

表單內需要讓申請人選擇申請類型：

```text
申請類型：
- 入職資源開通
- 在職資源新增
- 在職資源變更
- 離職資源回收
```

同一張表單可依不同申請類型顯示不同欄位與處理邏輯。

---

## 4. 為什麼需要這張電子表單

雖然人員入職時可以在人事建立人員資料時一併勾選資源，但仍然需要一張正式電子表單作為流程主單。

原因：

```text
1. 入職時需要留下正式申請與審批紀錄。
2. 在職期間可能新增或變更資源，仍需要有申請入口。
3. 離職時需要正式申請資源回收。
4. 工作單需要有來源主單可以追蹤。
5. HR、主管與各負責單位需要看到完整流程狀態。
```

因此，人員資源申請單可以有兩個入口：

```text
入口 1：人員入職流程內建立
入口 2：電子表單中心手動發起
```

### 4.1 入職流程內建立

HR 建立新人資料時，可以直接勾選需要申請的資源項目。系統背後自動建立一張「人員資源申請單」。

例如：

```text
HR 建立新人資料
→ 填寫姓名、公司、部門、職位、到職日、直屬主管
→ 勾選 Mattermost、Jira、門禁卡、指紋設定
→ 送出入職資源申請
→ 產生人員資源申請主單
```

### 4.2 電子表單中心手動發起

若員工入職一段時間後才需要某個資源，例如 Jira 或 VPN，則可以從電子表單中心手動發起「人員資源申請單」。

例如：

```text
電子表單 > 人員資源申請單
→ 申請類型：在職資源新增
→ 申請對象：王小明
→ 勾選 Jira、VPN
→ 填寫申請原因
→ 送出審批
```

---

## 5. 入職資源開通流程

入職時，HR 可以在人員資料建立流程中一併勾選資源需求。

建議流程：

```text
HR 建立新人資料
→ 勾選需要開通的資源項目
→ 送出入職資源申請
→ 直屬主管審批
→ 人事主管審批
→ 審批通過後產生各單位工作單
→ 各單位完成工作單
→ HR 確認整體完成
→ 主單結案
```

範例：

```text
新人：王小明
公司：代碼科技
部門：API 組
職位：後端工程師
直屬主管：Scott
到職日：2026-07-01

需要開通：
[x] Mattermost
[x] Jira
[x] 門禁卡
[x] 指紋設定
[ ] 電話
[ ] Apple ID
```

審批通過後產生：

```text
IT 工作單：開通 Mattermost
PMO / IT 工作單：開通 Jira
行政工作單：製作門禁卡
行政 / HR 工作單：設定指紋
```

---

## 6. 離職資源回收流程

離職時，系統應自動帶出該員工目前已綁定的資源，讓 HR 確認需要回收或停用的項目。

建議流程：

```text
HR 發起離職資源回收
→ 選擇離職人員
→ 系統帶出該員工已綁定資源
→ HR 確認回收項目
→ 直屬主管確認交接狀態
→ 人事主管確認
→ 審批通過後產生各單位回收工作單
→ 各單位完成回收
→ HR 確認全部完成
→ 主單結案
```

範例：

```text
離職人員：王小明
離職日：2026-07-31

系統帶出已綁定資源：
[x] Mattermost：wang.xiaoming
[x] Jira：xiaoming.wang
[x] 門禁卡：A123456
[x] 指紋：已設定
[x] Apple ID：xiaoming@company.com
[x] 電話分機：7088
```

離職時不應讓 HR 憑記憶勾選，而應由系統依人員資料中的資源綁定紀錄自動帶出。

## 7. 在職資源新增 / 變更流程

員工入職後，可能因為職務變更、加入新項目、工作內容調整，而需要新增或變更資源。

例如：

```text
入職時不需要 Jira。
加入 KX 項目後，需要 Jira 帳號。
```

此時應從電子表單中心發起「人員資源申請單」。

建議流程：

```text
員工 / 主管 / HR 發起申請
→ 選擇申請類型：在職資源新增 / 在職資源變更
→ 選擇申請對象
→ 勾選需要新增或變更的資源
→ 填寫申請原因
→ 直屬主管審批
→ 人事主管審批
→ 審批通過後產生工作單
→ 對應單位處理
→ 完成後回填結果並綁定人員資料
```

範例：

```text
申請類型：在職資源新增
申請對象：王小明
申請項目：Jira
申請原因：加入 KX 專案，需要處理 Jira 任務
```

---

## 8. 審批流程設計

人員資源申請單的預設審批流程建議為：

```text
直屬主管
→ 人事主管
```

### 8.1 直屬主管責任

直屬主管主要確認：

```text
1. 該員工是否確實需要這些資源。
2. 資源需求是否符合工作內容。
3. 在職新增或變更是否合理。
4. 離職時交接狀態是否確認。
```

### 8.2 人事主管責任

人事主管主要確認：

```text
1. 入職 / 離職 / 在職異動流程是否成立。
2. 人員資料是否正確。
3. 資源申請是否符合人事流程。
4. 是否可以進入後續工作單派發。
```

### 8.3 特殊資源的額外確認

第一版可以先以「直屬主管 → 人事主管」為主。

但設計上應保留未來擴充：部分特殊資源可能需要額外確認。

例如：

```text
Jira Admin 權限：可能需要 PMO 主管確認
VPN：可能需要 IT 主管確認
財務系統帳號：可能需要財務主管確認
高權限系統帳號：可能需要系統負責人確認
```

普通資源不應要求每個負責單位再做審批，避免流程過度複雜。

---

## 9. 工作單產生時機

工作單應在主表單審批通過後產生。

不建議表單一送出就產生工作單，原因是主管或人事主管尚未確認前，資源負責單位不應開始處理。

正確流程：

```text
人員資源申請單送出
→ 直屬主管審批
→ 人事主管審批
→ 審批通過
→ 系統依勾選資源項目產生工作單
→ 各單位並行處理工作單
```

## 10. 工作單中心設計

系統需要規劃「工作單中心」，用來讓各負責單位處理由人員資源申請單產生的任務。

工作單中心的定位是：

```text
接收待處理任務
顯示負責單位需要處理的工作單
回填處理結果
標記完成或退回
追蹤處理狀態
```

工作單中心不等於審批中心。

```text
審批中心：處理是否同意申請
工作單中心：處理實際執行任務
```

---

## 11. 工作單中心建議功能

建議工作單中心包含以下功能：

```text
工作單中心
├── 我的工作單
├── 我負責群組的工作單
├── 工作單查詢
└── 工作單設定
```

### 11.1 我的工作單

顯示目前登入者需要處理的工作單。

狀態可包含：

```text
待處理
處理中
已完成
無法處理
已退回
已取消
```

### 11.2 我負責群組的工作單

若使用者是某個工作單處理群組主管或管理者，可以查看該群組底下的工作單。

例如：

```text
台灣 IT 帳號處理群組
YL Jira 處理群組
日本行政門禁處理群組
```

### 11.3 工作單查詢

提供 HR、主管、工作單管理員或系統管理員查詢工作單。

可依以下條件查詢：

```text
人員
主申請單
資源項目
工作單類型
負責單位
處理群組
工作單狀態
地區
公司
項目
部門
日期
```

### 11.4 工作單設定

工作單設定用於管理：

```text
資源項目
工作單類型
工作單處理群組
工作單派發規則
```

此功能只開放給系統管理員或有管理權限的人員。

---

## 12. 工作單處理邏輯

每一張工作單至少需要支援以下操作：

```text
查看工作單內容
開始處理
回填處理結果
完成工作單
退回 / 無法處理
轉派
補充備註
查看操作紀錄
```

工作單完成時，負責單位可依資源類型回填資料。

例如：

### 12.1 帳號類資源

適用：Mattermost、Jira、Apple ID、VPN、Google Workspace 等。

可回填：

```text
帳號
Email
權限範圍
是否已啟用
是否已通知使用者
備註
```

### 12.2 實體或門禁類資源

適用：門禁卡、指紋設定、電話、分機、SIM 卡等。

可回填：

```text
卡號
分機號碼
設備編號
發放日期
設定日期
是否已交付
備註
```

### 12.3 完成後綁定人員資料

若工作單回填了帳號、卡號、分機等資料，系統應將結果綁定到人員資料中。

例如：

```text
人員資料 > 帳號 / 資源資訊
├── Mattermost：wang.xiaoming
├── Jira：xiaoming.wang
├── 門禁卡：A123456
├── Apple ID：xiaoming@company.com
├── 電話分機：7088
└── 指紋狀態：已設定
```

這樣離職時才能自動帶出需要回收的資源。

---

## 13. 主單狀態與工作單狀態

人員資源申請主單與工作單應有不同狀態。

### 13.1 主單狀態

建議主單狀態包含：

```text
草稿
審批中
已駁回
審批通過
工作單處理中
全部工作單完成
已結案
已取消
```

主單審批通過後，不代表流程完全完成，只代表可以進入工作單處理階段。

### 13.2 工作單狀態

建議工作單狀態包含：

```text
待派發
派發異常
待處理
處理中
已完成
無法處理
已退回
已取消
```

### 13.3 主單結案條件

主單需等所有必須完成的工作單完成後，才可結案。

例如：

```text
主單審批通過
→ 產生 5 張工作單
→ 4 張已完成，1 張待處理
→ 主單狀態：工作單處理中
→ 5 張全部完成
→ 主單狀態：全部工作單完成 / 已結案
```

---

## 14. 工作單派發規則

工作單中心需要有一個功能，用來設定不同資源項目應由哪些單位或人員處理。

建議功能名稱：

```text
工作單派發規則
```

或：

```text
工作單負責人設定
```

派發規則回答的是：

```text
哪一種資源，在某個地區 / 公司 / 項目 / 部門，應該派給哪個工作單處理群組？
```

---

## 15. 工作單處理群組

不建議派發規則直接指定到個人。

建議先建立「工作單處理群組」，再將人員加入群組。

例如：

```text
台灣 IT 帳號處理群組
台灣 PMO Jira 處理群組
日本行政門禁處理群組
英國財務系統帳號處理群組
YL 專案 Jira 處理群組
```

派發規則應指向處理群組，而不是直接指向個人。

原因：

```text
1. 人員異動時，只需調整群組成員。
2. 不需要修改大量派發規則。
3. 可支援多人共同處理。
4. 可支援群組主管查看整體工作單。
5. 權限管理較清楚。
```

---

## 16. 工作單派發需要依據的條件

工作單派發規則至少應支援以下條件：

```text
資源項目
地區
公司
事業部
項目
部門
負責單位
```

例如：

```text
資源項目：Jira
地區：台灣
項目：YL
處理群組：YL Jira 處理群組
```

```text
資源項目：Mattermost
地區：台灣
公司：代碼科技 / 天珤科技 / 天棨科技
處理群組：台灣 IT 帳號處理群組
```

```text
資源項目：門禁卡
地區：日本
公司：天元有限公司
處理群組：日本行政門禁處理群組
```

---

## 17. 工作單派發優先順序

當同一個資源項目可能符合多條派發規則時，系統需要有優先順序。

建議優先順序：

```text
1. 指定項目規則
2. 指定部門規則
3. 指定事業部規則
4. 指定公司規則
5. 指定地區規則
6. 全域預設規則
```

例如：

```text
Jira + YL 項目 → YL Jira 處理群組
Jira + 台灣 → 台灣 PMO Jira 處理群組
Jira + 全域 → 全域 PMO Jira 處理群組
```

如果申請人是 YL 項目人員，系統應優先派給 YL Jira 處理群組。

若找不到項目級規則，才往上找地區或全域規則。

---

## 18. 工作單可見權限

工作單中心的可見權限應依處理群組與角色決定。

### 18.1 一般處理人員

```text
若使用者是該工作單處理群組成員，則可以看到並處理該工作單。
```

例如：

```text
Kevin 屬於台灣 IT 帳號處理群組
→ Kevin 可以看到 Mattermost / Apple ID / VPN 類工作單
```

### 18.2 HR

HR 應可以查看人員入職、離職、資源異動主單底下所有工作單狀態。

HR 不一定能處理所有工作單，但需要追蹤整體進度。

### 18.3 直屬主管

直屬主管可以查看自己下屬相關主單與工作單進度。

主管不一定能看到敏感帳號細節，除非系統權限允許。

### 18.4 工作單管理員

工作單管理員可依其管理範圍查看與轉派工作單。

### 18.5 系統管理員

系統管理員可查看全部工作單與設定。

## 19. 工作單派發異常

若系統產生工作單時找不到對應派發規則，不可默默失敗。

系統應：

```text
1. 將工作單狀態標記為派發異常。
2. 通知 HR / 系統管理員 / 預設工作單管理群組。
3. 提示缺少哪個資源項目的派發規則。
4. 允許管理員手動指派處理群組。
5. 後續可補上派發規則，避免下次再次異常。
```

錯誤提示範例：

```text
無法派發 Jira 工作單：目前未設定「Jira / KX 項目」或其上層範圍的工作單處理群組。
```

---

## 20. 工作單轉派

工作單需要支援轉派。

常見原因：

```text
派錯群組
負責人請假
工作單應由其他單位處理
該資源已改由其他部門管理
```

轉派時必須記錄：

```text
操作人
原處理群組
新處理群組
轉派原因
轉派時間
```

轉派不應刪除原工作單紀錄，而是保留完整操作歷史。

## 25. 最終設計結論

本功能的最佳設計為：

```text
1. 不為 Mattermost、Jira、門禁卡、電話、指紋、Apple ID 各自建立電子表單。
2. 統一建立一張「人員資源申請單」。
3. 人員資源申請單支援入職、在職新增、在職變更、離職回收。
4. 表單內用勾選方式選擇需要申請或回收的資源。
5. 入職時可在人事建立人員資料時一併勾選資源，背後產生人員資源申請單。
6. 在職期間需要新增資源時，可從電子表單中心發起人員資源申請單。
7. 離職時系統自動帶出該員工已綁定資源，供 HR 確認回收。
8. 主表單只負責申請與審批，預設審批線為直屬主管與人事主管。
9. 審批通過後，系統依勾選資源自動產生各單位工作單。
10. 工作單由 IT、PMO、行政、HR、總務等對應單位處理。
11. 工作單處理人只看到自己處理群組負責的工作單。
12. HR 可追蹤主單底下全部工作單進度。
13. 工作單完成後可回填帳號、卡號、分機等資訊。
14. 回填結果需綁定到人員資料，方便後續查詢與離職回收。
15. 工作單中心需支援派發規則與處理群組設定。
16. 派發規則不直接綁定個人，而是指向工作單處理群組。
17. 若派發失敗，系統必須標記異常並通知管理人員。
```

一句話總結：

```text
電子表單決定要申請什麼，審批決定能不能申請，工作單中心決定誰來處理，處理結果回寫到人員資料。
```

## 26. 給 Claude Code 的開發指示

請依照本文件補充 OA 系統的人員資源申請與工作單中心功能。

開發時請注意：

```text
1. 不要為每一個資源項目建立獨立電子表單。
2. 建立一張統一的人員資源申請單。
3. 人員資源申請單需支援入職、在職新增、在職變更、離職回收。
4. 主單審批通過後才產生工作單。
5. 工作單需要依資源項目與人員組織資訊匹配處理群組。
6. 工作單處理群組成員才能看到對應工作單。
7. HR 可看到主單底下所有工作單狀態。
8. 工作單完成時需支援回填結果。
9. 帳號、卡號、分機等結果需回寫到人員資料。
10. 離職流程需自動帶出人員已綁定資源。
11. 若找不到工作單派發規則，需標記派發異常，不可靜默失敗。
12. 所有操作需保留紀錄，方便後續追蹤與稽核。
```
