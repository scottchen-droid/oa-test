## 1. 產品概覽

### 1.1 產品定位

**集團 OA 系統**是一套為跨國集團設計的企業管理平台，將財務申請、費用報銷、人事管理、出勤打卡、薪資發放整合在同一套系統內。

核心目標：

- 消除各地區各公司的資訊孤島
- 讓審批流程全程線上化、可追蹤
- 人事資料作為所有模組的帳號基礎

### 1.2 核心價值主張

| 使用者 | 痛點 | 我們的解法 |
|--------|------|------------|
| 員工 | 不知道申請進度、假期剩多少 | 即時狀態追蹤、假期餘額一目了然 |
| 主管 | 審批通知散落各通訊軟體 | 統一審批中心，一個地方處理所有待審 |
| 財務 | 報銷單跨公司難以拆帳 | 依地區 / 公司自動拆單，費用歸屬清楚 |
| HR | 員工資料維護複雜 | 單一員工主檔，入離職 / 薪資 / 假期統一管 |
| IT 管理員 | 權限設定分散難控管 | 超級管理員控制台，角色與資料範圍集中設定 |

### 1.3 系統邊界

**本系統包含：**
- 員工帳號管理（HR 統一維護）
- 財務 OA：採購申請、費用報銷
- 人事 OA：出勤、請假、薪資、績效
- 共用審批引擎
- 系統管理控制台

**本系統不包含：**
- 會計記帳系統（對接外部）
- 企業 ERP（可透過 API 對接）
- 即時通訊（可透過 Webhook 推送通知到 Slack / Teams）

## 2. 使用者角色（Personas）

### P1：一般員工（Employee）

**代表人物：** 陳大明，台灣業務部門工程師，每天需要打卡、偶爾申請採購或請假

**主要需求：**
- 手機或電腦快速打卡
- 查看自己的假期餘額
- 提交採購申請並追蹤進度
- 查看每月薪資單

**關鍵頁面：** 首頁儀表板、打卡、請假申請、採購申請、我的薪資

---

### P2：部門主管（Manager）

**代表人物：** 林主管，台灣研發部門主管，需要審批下屬的各類申請

**主要需求：**
- 統一查看所有待審事項
- 一鍵通過 / 駁回並填寫意見
- 查看部門出勤總覽
- 不想被各種通知打擾，但又不能漏審

**關鍵頁面：** 審批中心、部門出勤總覽

---

### P3：HR 管理員（HR Admin）

**代表人物：** 王小姐，集團 HR，負責員工入離職、薪資設定、假期政策

**主要需求：**
- 快速建立 / 更新員工資料
- 設定各公司假期政策與年資對應天數
- 查看全集團出勤異常
- 確認並發放每月薪資

**關鍵頁面：** 員工管理、假期政策、薪資計算、出勤總覽

---

### P4：財務人員（Finance Staff）

**代表人物：** 張會計，負責審核報銷單、確認付款、對帳

**主要需求：**
- 查看所有待審核報銷單
- 確認單據是否完整、金額是否正確
- 依地區 / 公司匯出報表
- 標記已付款

**關鍵頁面：** 報銷審核、財務報表、付款確認

---

### P5：超級管理員（Super Admin）

**代表人物：** 系統 IT 負責人，負責帳號開通、權限設定、系統維護

**主要需求：**
- 開通 / 停用員工帳號
- 替員工分配角色與資料範圍
- 查看所有系統操作紀錄
- 設定審批流模板

**關鍵頁面：** 帳號管理、角色權限、審批流設定、稽核日誌

## 4. 功能模組規格

### 4.1 帳號與認證模組

#### 功能清單

| 功能 | 使用者 | 優先級 |
|------|--------|--------|
| 工號 + 密碼登入 | 所有人 | P0 |
| 帳號首次啟用必須修改密碼 | 所有人 | P0 |

#### 關鍵規則

- 帳號由 HR 建立，員工不可自行註冊
- 離職員工帳號狀態改為 `resigned`，無法登入，但資料保留
- 超級管理員可強制重置任何帳號密碼
- 密碼需符合：8 字元以上、含大小寫與數字

---

### 4.2 首頁儀表板模組

#### 整體 Layout

首頁採用**三欄式 Layout**（Top Header + Left Sidebar + Main Content + Right Panel）。

```
┌──────────────────────────────────────────────────┐
│ Top Header (Logo / 通知 / 使用者資訊)              │
├────────────┬───────────────────────┬─────────────┤
│ Left (220px)│ Main Content         │ Right (300px)│
│ 快捷入口   │ 分頁：總覽/個人/主管  │ OA 小助手   │
│ 今日提醒   │ 公告列表              │ 登入資訊卡  │
│ 明日提醒   │                       │ Banner      │
│ 最近使用   │                       │             │
└────────────┴───────────────────────┴─────────────┘
```

#### Left Sidebar — 快捷入口

| 功能 | Route | Badge |
|------|-------|-------|
| 通知 | `/notifications` | 未讀數 |
| 工作 | `/approvals` | 待審數 |
| 簽核 | `/approvals` | 待簽數 |
| 請假 | `/leaves` | — |
| 公告 | `/announcements` | 未讀數 |
| 考勤 | `/attendance` | — |

- Badge 大於 99 顯示 `99+`
- 無權限的入口自動隱藏
- 點擊後記錄最近使用功能

#### Left Sidebar — 今日 / 明日提醒

每筆提醒欄位：`id / title / type / dueDate / dueTime / route / priority`  
無資料時顯示「無提醒事項」

#### Left Sidebar — 最近使用功能

最多顯示 5 筆，依 `lastUsedAt` 由新到舊排序。

#### Main Content — 主分頁

| 分頁 | 權限 | 初版狀態 |
|------|------|----------|
| 總覽 | 所有人 | 實作 |
| 個人 | 所有人 | Placeholder |
| 主管 | 主管角色 | Placeholder |
| 資訊中心 | 所有人 | Placeholder |

#### Main Content — 公告列表

**欄位：** id / title / category / type / authorName / publishDate / contentPreview / attachments / tags / isPinned / readStatus

**排序：** 置頂優先 → publishDate 由新到舊  
**顯示：** 最新 5 筆，右下角 More 連結

**附件（AnnouncementAttachment）：**  
```ts
type Attachment = { id, fileName, fileUrl, fileType, fileSize? }
```
點擊附件需驗證存取權限，不可暴露未授權 URL。

**公告可見範圍（scopeType）：** global / company / department / role

#### Right Panel — 登入資訊卡

顯示：userName / ip / companyName / companyCode / employeeNo / departmentName / positionName / roleName  
右上角登出按鈕，點擊需二次確認。

#### Right Panel — OA 小助手 / Banner

初版做 UI placeholder，不需要真的串接 AI 或後端 Banner API。

#### Dashboard API Response

單一 API `GET /api/dashboard` 回傳：

```ts
type DashboardResponse = {
  currentUser: CurrentUser          // 使用者 + 組織資訊
  shortcuts: ShortcutItem[]         // 快捷入口（含 badgeCount）
  todayReminders: ReminderItem[]    // 今日提醒
  tomorrowReminders: ReminderItem[] // 明日提醒
  recentFunctions: RecentFunctionItem[]
  announcements: AnnouncementItem[]
}
```

#### 狀態處理

| 狀態 | 說明 |
|------|------|
| Loading | Skeleton 佔位（快捷入口、提醒、公告、登入資訊） |
| Empty | 各區塊顯示對應空狀態文案 |
| Error | 顯示「資料載入失敗，請稍後再試」+ 重新載入按鈕 |

#### Widget 清單（依角色）

| Widget | 顯示對象 | 說明 |
|--------|----------|------|
| 快捷入口 Badge | 所有人 | 待審批件數、未讀通知 |
| 今日提醒 | 所有人 | 今日待辦、請假、出勤異常 |
| 公告列表 | 所有人 | 依可見範圍過濾 |
| 登入資訊卡 | 所有人 | 公司、部門、工號、角色 |
| 主管分頁 | 主管 | 部門待審總覽（Phase 2） |
| 出勤異常提醒 | HR Admin | 今日缺勤/遲到人數（Phase 2） |
| 本月薪資預覽 | 所有員工 | 薪資確認後顯示（Phase 2） |

#### 功能清單

| 功能 | 說明 | 優先級 |
|------|------|--------|
| 查看假期餘額 | 各假別剩餘天數 | P0 |

- 特休依年資於年初配給（或按月累積，待確認）

### 4.5 採購申請模組

#### 申請單狀態流轉

```
草稿 → 審批中 → 已通過 → 已結案
             ↘ 已駁回 → 草稿（修改後重送）
                        ↘ 已取消
```

#### 功能清單

| 功能 | 說明 | 優先級 |
|------|------|--------|
| 建立採購申請 | 填寫基本資訊、費用分攤明細 | P0 |
| 儲存草稿 | 未完成時可儲存 | P0 |
| 送出申請 | 觸發審批流 | P0 |
| 查看申請進度 | 審批流時間軸 | P0 |
| 撤回申請 | 審批中可撤回到草稿 | P1 |
| 費用分攤明細 | 依地區 / 公司 / 部門 / 專案分攤 | P0 |
| 附件上傳 | 報價單、說明文件 | P0 |
| 複製申請單 | 快速建立相似申請 | P2 |

#### 申請單欄位規格

**基本資訊**
- 申請標題（必填）
- 申請說明（必填）
- 幣別（下拉，必填）
- 總金額（必填，由分攤明細加總）
- 附件（可多份）

**費用分攤明細**（可新增多筆）
- 地區（必填）
- 公司（必填，依地區篩選）
- 部門（選填）
- 事業部（選填）
- 專案（選填）
- 金額（必填）
- 說明（選填）

## 6. 主要用戶旅程

### 6.1 員工採購申請旅程

```
員工登入
  ↓
首頁儀表板（看到快捷採購按鈕）
  ↓
新增採購申請（填寫基本資訊 + 費用分攤）
  ↓
送出申請
  ↓
系統自動匹配審批流模板，產生審批實例
  ↓
各審批人收到通知 → 逐節點審批
  ↓
全部通過 → 申請人收到通過通知
  ↓
申請人 / 財務人員建立報銷單
  ↓
報銷審批 + 財務審核 → 付款完成
```

### 6.2 員工請假旅程

```
員工登入
  ↓
點擊「申請請假」
  ↓
系統顯示假期餘額 → 員工選擇假別、日期
  ↓
系統自動計算扣除天數（排除假日）
  ↓
員工填寫原因、選擇代理人
  ↓
送出 → 主管收到審批通知
  ↓
主管通過 → 假期餘額更新、出勤紀錄更新
  ↓
員工收到通過通知
```

### 6.3 HR 月度薪資發放旅程

```
HR 進入薪資管理
  ↓
建立月度薪資期間
  ↓
系統自動計算（底薪 + 津貼 + 加班費 - 扣項）
  ↓
HR 逐一核對或批量核對
  ↓
HR 確認薪資（需輸入確認文字）→ 薪資期間鎖定
  ↓
員工可查看薪資單
  ↓
HR 標記已發薪 → 流程完成
```

### 6.4 新員工入職旅程

```
HR 在系統建立員工資料
  ↓
系統自動建立使用者帳號（status = pending_activation）
  ↓
系統發送啟用 Email 給員工
  ↓
員工點擊連結，設置密碼
  ↓
帳號啟用成功（status = active）
  ↓
Super Admin / HR Admin 分配角色
  ↓
員工可正常使用系統
```
